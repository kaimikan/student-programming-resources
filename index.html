Git — кратък курс

Git пази историята на кода ви и позволява няколко души да работят по един проект, без да си пращат архиви по имейл. GitHub е сайтът, където живее споделеното копие (remote).

1. Настройка (веднъж на всеки компютър)

Ако git го няма на компютъра — изтеглете го от git-scm.com/downloads (на Windows инсталацията включва и Git Bash терминал; проверявате с git --version). После се представете:

git config --global user.name "Иван Иванов"
git config --global user.email "ivan@example.com"

2. Взимане на проекта

# съществуващ проект от GitHub:
git clone https://github.com/vashiat-ekip/proekt.git

# нов проект от нулата:
git init
git remote add origin https://github.com/vashiat-ekip/proekt.git

3. Ежедневният цикъл

git pull                          # 1. взимате промените на колегите (винаги първо!)
#    ...работите...
git status                        # 2. какво сте променили
git add .                         # 3. избирате промените
git commit -m "Форма за вход"     # 4. запис в историята с кратко описание
git push                          # 5. качвате в GitHub

Правила за екипа: pull преди да започнете, commit-вайте често (по-добре 10 малки, отколкото 1 огромен), пишете съобщения, които казват какво е направено.

4. Същият цикъл, но с бутони — VS Code

Всичко от горния цикъл го има и в Source Control панела на VS Code (иконата с разклонението вляво): промените се виждат като списък, add е „+“ до файла, commit е полето за съобщение + бутонът Commit, а pull/push е бутонът Sync Changes. При конфликт VS Code отваря удобен редактор с бутони Accept Current / Incoming / Both — по-лесно от търсенето на маркерите на ръка. Подробно: Using Git source control in VS Code.

Ползвайте UI-а свободно, но знайте кои команди натиска той вместо вас — когато нещо се обърка, решението почти винаги минава през терминала. Затова първо разберете цикъла от т. 3.

5. Конфликти

Ако двама сте пипали един и същи ред, при git pull Git ще маркира файла с <<<<<<<, =======, >>>>>>>. Отворете файла, оставете верния вариант, изтрийте маркерите и направете git add . + git commit. Не е страшно — случва се на всички.

6. .gitignore — какво НЕ се качва

Създайте файл .gitignore в корена на проекта още в началото:

node_modules/
__pycache__/
*.pyc
.env
.DS_Store

Никога не качвайте пароли и API ключове в Git. Дръжте ги във файл .env, който е в .gitignore. Веднъж качен, ключът остава в историята завинаги (дори да го изтриете после) и ботове го намират за минути — тогава единственото решение е да го смените от сайта на услугата.

7. Работа от различни (споделени) компютри

Вие не работите всеки ден на една и съща машина, а компютрите са споделени. Проблемът: ако запазите паролата/токена си на диска, следващият човек на този компютър може да пише в GitHub от ваше име. Ето безопасната схема:

  1. Направете си fine-grained token вместо парола: в GitHub → Settings → Developer settings → Fine-grained personal access tokens. Ограничете го само до repo-то на проекта, права „Contents: Read and write“, и срок на валидност до края на програмата. Този token се използва вместо парола при git push. Ако изтече ключът — правите нов за 2 минути, а откраднат token не дава достъп до целия ви акаунт.
  2. Кажете на Git да помни токена само в паметта, не на диска: 
    git config --global credential.helper 'cache --timeout=28800'
    Токенът се пази 8 часа в RAM и изчезва при рестарт — не остава файл на диска. (Ако някой преди вас е ползвал store, проверете и изтрийте: rm -f ~/.git-credentials.)
  3. В края на деня: излезте от github.com в браузъра (или изобщо влизайте само в частен/incognito прозорец), не позволявайте на браузъра да запази паролата, и излезте от сесията на компютъра.

Алтернатива без нищо локално: GitHub Codespaces — цял VS Code в браузъра, закачен за repo-то ви. Нищо не се пази на споделения компютър, а безплатният лимит е 120 core-часа на месец. Влизате от кой да е компютър само с GitHub акаунта си.

8. Клонове (branches) — ако ви остане време

За двуседмичен спринт е приемливо всички да работят в main с често pull-ване. Ако екипът е 3+ души и се настъпвате, минете на клонове:

git switch -c nova-funkcia     # нов клон
# ...работите, add, commit...
git push -u origin nova-funkcia
# после в GitHub: Pull Request → Merge

emailjs.com →

Защо писмата ви отиват в Spam?

Няколко екипа вече се сблъскаха с това. Най-честите причини, подредени по вероятност:

  1. Изпращате през личен Gmail (SMTP/nodemailer). Gmail не е за автоматични писма — има строги лимити и писмата от скрипт изглеждат като спам. Ползвайте услуга от горните.
  2. Липсват SPF, DKIM и DMARC записи. Това са три DNS записа, които доказват, че вашият сървър има право да изпраща от името на домейна ви. Без тях получаващият сървър няма как да ви разпознае и ви праща в спам. Когато „верифицирате домейн” в Resend/Brevo, те ви дават наготово точно тези записи — просто ги добавяте в DNS-а на домейна си.
  3. Нямате собствен домейн. От споделения тестов домейн на услугата (напр. onboarding@resend.dev) доставимостта е слаба. Истинската доставимост започва със собствен домейн (вижте секция Домейни).
  4. Съдържанието изглежда спам: САМО ГЛАВНИ БУКВИ, много удивителни, куп линкове, само една голяма картинка без текст.
  5. Нов домейн без репутация. Първите дни изпращайте малко и постепенно — репутацията се гради.

Проверете се: пратете писмо до mail-tester.com — дава ви оценка от 10 и списък какво точно ви куца (SPF? DKIM? съдържание?).

Изпращане на SMS

SMS е осезаемо по-скъп и по-регулиран от имейла. За MVP демо trial план стига, но прочетете ограниченията, за да не ви изненадат на демо деня.

Индустриалният стандарт. Trial акаунтът дава ~$15 кредит, но: изпраща само до 5 предварително верифицирани номера, максимум 50 съобщения/ден, и всеки SMS започва с „Sent from a Twilio trial account“.

Vonage Trial

Основната алтернатива на Twilio, с €2 начален кредит и подобни trial ограничения (само верифицирани номера). API-то е също толкова просто.

vonage.com →

core.telegram.org/bots →

Най-простото възможно API (една HTTP заявка), с 1 безплатен SMS на ден за тестове. За реална употреба се плаща на бройка.

Извличане на данни (web scraping)

Когато MVP-то ви има нужда от съдържание, което вече съществува на друг сайт — обяви, цени, статии, разписания — можете да го извлечете автоматично. Ето как да подходите, от най-лесното към най-сложното.

Златното правило: първо проверете за API, после мислете за парсене на HTML. Отворете сайта, натиснете F12 → таб Network → презаредете страницата и филтрирайте по „Fetch/XHR“. Много често съдържанието пристига като чист JSON от вътрешно API — и една заявка към него е десетократно по-лесна и по-стабилна от ровене в HTML. Този навик струва повече от всяка библиотека по-долу.

Jina Reader Безплатно

Най-бързият начин за „копирам линк → взимам съдържанието“: слагате r.jina.ai/ пред който и да е URL и получавате страницата като чист текст/markdown. Без код и без ключ за малки обеми — идеално за MVP.

jina.ai/reader →

Firecrawl Free tier

URL → структуриран JSON или markdown през API; може да обходи и цял сайт. По-подреденият вариант, когато ви трябват конкретни полета (цена, заглавие, дата). Безплатните кредити стигат за проект.

firecrawl.dev →

requests + BeautifulSoup Безплатно

Класиката в Python за статични страници: смъквате HTML-а и вадите елементите по CSS селектор. Селектора го взимате наготово от браузъра: десен бутон върху елемента → Inspect → десен бутон в кода → Copy selector. В Node еквивалентът е fetch + cheerio.

beautifulsoup docs →

Playwright Безплатно

Управлява истински браузър — за страници, които се строят с JavaScript и през requests изглеждат празни. По-бавен и по-тежък, затова е вторият избор, не първият.

playwright.dev →

Уловката, в която всички стъпват: scraping не става от frontend JavaScript — браузърът блокира заявките към чужди сайтове (CORS). Извличането трябва да се случи на сървъра ви (или през Jina/Firecrawl като посредник), а frontend-ът да пита вашия сървър.

Добри обноски (и самосъхранение): погледнете robots.txt и условията на сайта; слагайте пауза между заявките и кеширайте резултатите — стотици заявки в секунда водят до блокиран IP точно преди демото; и внимавайте с личните данни — извлечени имена и имейли на реални хора са GDPR територия, не материал за MVP.

Backend (Python, Node и т.н.)

Първо се запитайте: трябва ли ви изобщо собствен backend? Ако нуждите ви са регистрация/вход на потребители и записване и показване на съдържание — това е точно каквото Supabase или Firebase дават наготово: говорите с тях директно от frontend-а, с вградени правила кой какво може да чете и пише, и няма сървър за поддръжка и събуждане. Собствен сървър (Flask/FastAPI/Express) си заслужава, когато имате истинска сървърна логика: интеграции с външни API-та с тайни ключове, плащания, по-тежки изчисления.

pythonanywhere.com →

Railway Платено

Fly.io Платено

Докер контейнери близо до потребителите, заплащане на употреба (малко приложение излиза центове). За по-напреднали екипи.

Две уловки на безплатния backend хостинг: (1) Преди демо — отворете си приложението 5 минути по-рано, за да се събуди, иначе публиката гледа зареждащ екран. (2) Файловата система е временна — качени файлове и SQLite бази изчезват при всеки deploy/рестарт. За данни ползвайте истинска база от секция Бази данни.

Персистентни бази данни

Как се свързва домейн с Vercel/Netlify

  1. В таблото на Vercel: Project → Settings → Domains → добавяте vashproekt.me.
  2. Vercel ви показва какви DNS записи да сложите (обикновено един A запис за домейна и CNAME за www).
  3. Добавяте ги в DNS панела на регистратора. До час домейнът работи, SSL сертификатът (https) се издава автоматично и безплатно.

Колко струва да поддържате проекта след програмата?

  • Статичен сайт / frontend: 0 лв. — безплатните планове на Vercel/Netlify/Cloudflare са достатъчни за години напред. Плащате само домейна (~20–25 лв/год).
  • С backend и база: ~$5–7/мес (Railway Hobby или Render Starter, за да не заспива) + домейн. Supabase free tier често остава достатъчен.
  • За любопитните: VPS от Hetzner (~€4/мес) — цял собствен сървър, но всичко си го поддържате сами. Чудесен начин да се научи, лош начин да се спази срок.
